# 前后台内容映射与首页信息架构计划

## 1. 目标

当前问题不是“后台数据有没有接上”，而是“后台素材”和“前台叙事”之间缺少翻译层。

后台应该作为素材库，前台负责筛选、排序和表达。首页不应该逐字段搬运后台数据，而应该围绕访问者的理解路径组织内容：

1. 3 秒内知道你是谁。
2. 30 秒内理解你为什么转向 AI 产品 / 运营。
3. 1-2 分钟内看到可信的项目和学习证据。
4. 愿意进一步查看项目详情、联系你或索取简历。

## 2. 前台页面分工

### 首页

首页只承担“建立认知 + 建立信任 + 引导深入”的任务。

建议保留：

- Hero：身份、方向、一句话定位。
- 当前定位：建筑学背景、AI 工具学习、产品意识、实践方式。
- 核心项目：1 个主项目 + 最多 2 个辅助项目。
- 学习路径摘要：展示持续学习轨迹，不展开成证书墙。
- AI 理解：展示判断力和方法论。
- 联系区：联系方式、简历说明、下一步行动。

建议弱化或移出首页：

- 全部项目列表。
- 全部技能列表。
- 全量留言列表。
- 简历生成系统入口。
- 过多后台字段原样展示。

### 项目详情页

项目详情页负责承接完整信息。

适合展示：

- 项目名称、角色、周期。
- STAR 内容：情境、任务、行动、结果。
- 关键词。
- 图片或过程截图。
- GitHub / Vercel / 作品链接。
- 项目复盘：做对了什么、还差什么、下一步迭代。

### 管理后台

后台负责维护素材，不直接决定首页长什么样。

后台重点能力：

- 管理个人信息。
- 管理技能标签。
- 管理项目素材。
- 管理教育 / 学习经历。
- 管理留言审核。
- 管理简历生成。

## 3. 后台模块到前台模块映射

| 后台模块 | 核心字段 | 首页展示方式 | 详情页 / 后台用途 | 优先级 |
| --- | --- | --- | --- | --- |
| Profile 个人信息 | name, title, slogan, contact | Hero、导航、页脚、联系区 | 联系方式脱敏、简历基础信息 | P0 |
| Skills 技能 | name, category, weight | 当前定位区展示前 6-8 个高权重标签 | 简历匹配、能力分类、详情补充 | P0 |
| Projects 项目 | name, role, time, STAR, keywords, isPublic, sortOrder | 首页展示 1 个主项目 + 2 个辅助项目 | 项目详情页完整展开 | P0 |
| Education 教育 | school, major, time, description | 首页只作为建筑学背景的一句话证据 | 简历系统、关于页、详情补充 | P1 |
| Portfolio 外链 | title, url, type | 不单独铺开，跟项目卡片绑定 | 项目详情页外链 | P1 |
| Comments 留言 | nickname, content, status, reply | 首页只保留轻量入口或精选 1-2 条 | 留言审核、回复、管理 | P2 |
| Resume 简历记录 | JD, projectOrder, customContent | 首页不主打，联系区弱入口 | 后台生成、预览、保存 | P1 |

## 4. 首页模块具体接法

### 4.1 Hero

数据来源：

- `profile.name`
- `profile.title`
- `profile.slogan`

展示规则：

- `h1` 保持主叙事，不频繁受后台影响。
- `profile.title` 用作职业方向。
- `profile.slogan` 用作副文案。
- 如果后台为空，使用默认转型叙事兜底。

不展示：

- 联系方式明文。
- 大段个人经历。
- 简历系统。

### 4.2 当前定位

数据来源：

- 静态叙事卡片。
- `skills` 高权重标签。

展示规则：

- 技能不做“技能墙”，只作为定位证据。
- 取 `weight` 最高的 6-8 个技能。
- 如果有 `category`，后续可按 `AI 工具 / 产品能力 / 内容能力 / 工程协作` 分组。

后续优化：

- 后台技能分类建议统一为：
  - AI 工具
  - 产品能力
  - 内容与运营
  - 工程协作
  - 建筑学迁移能力

### 4.3 核心项目

数据来源：

- `projects` 中 `isPublic = true` 的项目。
- 优先使用 `sortOrder` 最靠前的项目。
- 若项目名称或关键词包含“个人主页 / homepage”，优先作为主项目。

展示规则：

- 首页只展示 1 个主项目。
- 最多展示 2 个辅助项目作为证据。
- 完整 STAR 内容不在首页展开，点击进入详情页。

建议后台填写规则：

- 想作为首页主项目：项目名或关键词加入“个人主页”。
- 想展示在首页：打开 `isPublic`。
- 想调整顺序：调整 `sortOrder`。

### 4.4 学习路径

当前状态：

- 暂时适合保持静态。
- 因为它更像个人叙事，不一定适合用旧项目字段硬套。

后续建议：

- 新增 `learning_records` 表。
- 字段建议：`title`, `type`, `time`, `role`, `output`, `reflection`, `sortOrder`, `isPublic`。
- 用于记录 n8n、Coze、短剧、Vibecoding、OpenClaw、训练营志愿者等经历。

状态：

- 已开始执行，后台已新增“学习经历”管理入口，首页学习路径会优先读取公开学习经历，没有数据时使用静态兜底。

### 4.5 AI 理解

当前状态：

- 暂时保持静态或半静态。
- 它负责表达判断力，不是经历列表。

后续建议：

- 可以新增 `insight_cards` 表。
- 字段建议：`title`, `content`, `sourceProjectId`, `sortOrder`, `isPublic`。
- 用项目经历反哺观点，而不是单独堆观点。

状态：

- 已开始执行，后台已新增“观点卡片”管理入口，首页 AI 理解会优先读取公开观点卡片，没有数据时使用静态兜底。

### 4.6 联系区

数据来源：

- `profile.contact.email`
- `profile.contact.wechat`
- `skills` 少量能力标签

展示规则：

- 联系方式默认脱敏。
- 点击复制真实值。
- 不在 HTML 中直接显式展示明文。
- 简历系统只做弱提示，不作为首页主功能。

## 5. 后台需要补强的字段

短期不必改数据库，先用现有字段约定。

现有字段约定：

- `project.isPublic`：是否展示。
- `project.sortOrder`：首页和详情排序。
- `project.keywords`：用于标记“个人主页”“AI 产品”“n8n”等主题。
- `skill.weight`：首页优先级。
- `skill.category`：能力分组。

中期建议新增：

| 新字段 / 新表 | 作用 | 优先级 |
| --- | --- | --- |
| `projects.featured` | 明确首页主项目，替代关键词约定 | P1 |
| `projects.summary` | 首页短摘要，避免用 STAR 字段硬截断 | P1 |
| `learning_records` | 管理学习路径和训练营经历 | P1 |
| `insight_cards` | 管理 AI 理解 / 方法论卡片 | P2 |
| `portfolio.projectId` | 外链绑定具体项目 | P2 |

## 6. 下一步执行顺序

### 第一步：首页减法与融合

目标：

- 首页不再像后台数据列表。
- 数据只作为叙事证据出现。

任务：

- 保留 Hero、当前定位、核心项目、学习路径、AI 理解、联系区。
- 当前定位接技能标签。
- 核心项目接主项目和辅助项目。
- 移除独立“全部技能 / 全部项目”展示。

状态：

- 已开始执行，当前版本已将技能和项目嵌入对应叙事模块。

### 第二步：项目详情页升级

目标：

- 首页少讲，详情页讲清楚。

任务：

- 优化项目详情页视觉。
- 按 STAR 分块展示。
- 增加“项目复盘 / 下一步迭代”区域。
- 展示项目图片和外链。

状态：

- 已开始执行，项目详情页已支持 `summary`、`reflection`、`githubUrl`、`demoUrl`、`portfolioUrl`。

### 第三步：后台内容填写规范

目标：

- 让后台填写后能稳定影响前台。

任务：

- 给项目表单增加填写提示。
- 给技能分类提供推荐选项。
- 给首页主项目添加明确说明。

状态：

- 已开始执行，项目表单已支持“首页短摘要”“设为首页主项目”“项目复盘”“项目外链”。
- 技能表单已加入推荐分类：AI工具 / 产品能力 / 内容运营 / 工程协作 / 建筑学迁移能力。

### 第四步：简历系统升级

目标：

- 简历系统继续作为后台能力，而不是首页主卖点。

任务：

- JD 关键词匹配。
- 项目自动排序。
- 简历风格预设。
- 可编辑预览。
- PDF 导出预留或实现。

## 7. 判断标准

每次改动都用下面 4 个问题检查：

1. 这个内容是否帮助访问者更快理解“我是谁”？
2. 这个内容是否证明我具备 AI 产品 / 运营相关潜力？
3. 这个内容是否适合放首页，还是应该去详情页？
4. 这个字段是原样展示，还是经过了前台叙事翻译？

如果答案不清楚，就不要直接塞到首页。
